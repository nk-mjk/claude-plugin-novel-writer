# novel-writer

> 멀티 에이전트 소설 집필 플러그인 — 한 번의 요청으로 *설계 → 집필 → 대사 입체화 → 검수 → 합본*까지.
> 등록된 5개 서브에이전트를 오케스트레이터가 이름으로 호출해 분업한다. Blue × Claude.
>
> **버전 0.4.3** · 스킬 1 + 등록 에이전트 5 + 조립 스크립트 1

---

## 무엇을 하나

소설 제목·장르·소재·화수만 입력하면, 오케스트레이터(스킬)가 전체 작업을 단계별로 지휘한다.
한 명의 거대한 작가가 통째로 쓰는 대신, **역할이 다른 에이전트들이 분업**한다 — 뼈대를 세우는 설계자, 화별로 동시에 쓰는 집필가들, 인물을 연기하는 배우들, 흐름과 사실을 따로 보는 검수자들. 합치기는 LLM이 아니라 스크립트가 한다.

결과물은 합본 한 편(`novel-final.md`)과, 처음 쓰는 사람을 위한 폴더 안내(`0_먼저읽으세요.md`)다.

---

## 구성

- **스킬 1개** — `novel-writer` (오케스트레이터): Phase 0~4 흐름을 진행한다.
- **등록 에이전트 5개**:

  | 호출 이름 (`subagent_type`) | 역할 | 실행 | 출력 |
  |---|---|---|---|
  | `novel-writer:story-arc` | 전체 뼈대(세계관·인물·아크) 설계 | 순차 (STEP 1) | `story-bible.md` |
  | `novel-writer:episode-writer` | 에피소드 집필 | 병렬 (STEP 2 ×N) | `ep0X.md` |
  | `novel-writer:character-actor` | 배역 대사 연기(대사 입체화) | 선택 (STEP 2.5, 배역별) | 입체 대사 반환(메인이 교체) |
  | `novel-writer:flow-checker` | 흐름·연속성·감정선 검수 | 병렬 (STEP 3) | `review-flow.md` |
  | `novel-writer:fact-checker` | 사실·세계관 규칙 검수 | 병렬 (STEP 3) | `review-facts.md` |

- **조립 스크립트** — `skills/novel-writer/scripts/build_novel.py`: Phase 4에서 검수 반영본 `ep0X.md`를 결정적으로 합쳐 `novel-final.md`를 만든다 (LLM 토큰 0, 제목·장르·화수 자동 감지).

## 작동 흐름

```
Phase 0 ── 입력 수집 (제목·장르·소재·화수)                         AskUserQuestion
              ↓ input.md

Phase 1 ── [story-arc]  ───────────────────────────────────────  순차
              ↓ story-bible.md (세계관·인물·아크 — 유일한 공유 레일)

Phase 2 ── [episode-writer #1] [#2] … [#N]  ───────────────────  병렬 ×N
              ↓ ep01.md … ep0N.md

Phase 2.5  [character-actor 배역별]  ──────────────────────────  선택
              ↓ 입체 대사로 해당 부분만 교체 (교체 전 ep0X.orig.md 백업)
              recast = 병렬 / react(핑퐁) = 순차

Phase 3 ── [flow-checker]  [fact-checker]  ────────────────────  병렬
              ↓ review-flow.md · review-facts.md

Phase 4 ── 검수 반영(오케스트레이터, Edit) + 조립(build_novel.py)  판단=LLM / 조립=코드
              ↓
           novel-final.md  +  0_먼저읽으세요.md
```

각 단계 전환 시 오케스트레이터는 **진행 배너**를 코드블록 한 줄로 띄워, 일반 설명과 구별되게 현재 위치를 알린다:

```
********** 스토리 바이블 완료 — 제목: {제목} **********
```

## 연기 패스 (STEP 2.5) — 대사를 입체적으로

> **v0.4에서 추가된 단계.** novel-writer의 가장 큰 도약이다.

집필 에이전트 한 명이 모든 인물을 1인 다역으로 쓰면, 등장인물의 말투가 서로 닮아 **대사가 평면적으로 수렴**하기 쉽다. 연기 패스는 그 초고 위에 **배역 에이전트(`character-actor`)가 자기 배역만 다시 연기**하는 층을 얹어 대사를 살린다. 초고는 흐름을 고정하는 *레일*, 액터는 인물의 *목소리* — 영화 후시녹음(ADR)에서 감독이 배우에게 장면을 다시 연기시키는 것과 같다. 각 배우는 자기 **쪽대본(그 인물이 아는 정보까지만)**을 받으므로, 인물 사이의 정보 비대칭과 긴장도 함께 살아난다.

대사 비중이 낮거나 빠른 진행이 필요하면 건너뛴다. 두 모드가 있고, 오케스트레이터(감독)가 장면 등급으로 고른다:

- **recast (재연기)** — 일반 대사를 인물 목소리로 다시 쓰기. 배역별 **독립 호출 → 병렬**. 빠르고 싸다. 대부분의 대사는 이걸로 충분하다.
- **react (핑퐁)** — 논쟁·대결·감정의 정점 장면. 감독이 A→B→A로 **턴을 중계하는 순차 호출**. 느리고 비싸지만, 대사가 서로 받아치며 비유와 펀치라인이 양쪽 입을 오가며 누적·전이된다.

배역끼리 직접 대화하지 못하므로(컨텍스트 격리), 감독이 턴을 중계하고, 교체는 본문의 **해당 대사 부분만** 한다(전체 재작성 금지). 실측에선 **기능적으로 쓰기 쉬운 단역일수록 입체화 효과가 컸고**, 입체화가 흐름을 흔들지 않았는지는 이어지는 검수(STEP 3)가 받쳐준다.

## 사용법

```
"소설 써줘"   또는   "novel-writer 시작"
```

스킬이 입력(제목·장르·소재·화수)을 받아 위 흐름대로 진행한다. 화수는 **1~5화 권장** — 초과 시 컨텍스트 소비가 커지므로 먼저 확인한다.

## 결과물

작업이 끝나면 `{소설제목_날짜}/` 폴더에 다음이 생긴다:

- **`novel-final.md`** — 완성된 소설 합본. 보통 이 파일 하나면 충분하다.
- **`0_먼저읽으세요.md`** — 폴더 안내. 어떤 파일이 무엇이고 무엇을 지워도 되는지 자동 정리(파일이 많아 보여 당황하지 않게). `build_novel.py`가 함께 생성한다.
- `ep01.md … ep0N.md` — 화별 원고(합본에 이미 포함).
- `story-bible.md` — 세계관·인물·아크 설계서(이어 쓸 때 필요).
- `input.md`, `review-*.md`, `ep0X.orig.md` — 작업 과정 파일(지워도 됨).
- `.agent_log.md` — 에이전트별 토큰·도구·시간·`agentId` 로그(숨김, 분석용).

## 예제

`examples/Before_After_AI/`에 이 플러그인으로 처음부터 끝까지 만든 단편 SF가 들어 있다 — **「Before/After AI」(SF · 5화 완결)**. 2226년의 초지능이 200년 전 'AI 여명기'를 회고하는 액자식 구조의 작품으로, 입력(`input.md`)부터 설계(`story-bible.md`), 화별 원고(`ep01~05.md`), 검수 기록(`review-*.md`), 합본(`novel-final.md`)까지 한 회차의 전 과정이 그대로 담겨 있다.

`_agent_log.md`에는 13번의 에이전트 호출(설계 1 · 집필 5 · 연기 5 · 검수 2)이 토큰·소요 시간과 함께 기록돼 있어, 멀티 에이전트가 실제로 분업한 흔적을 확인할 수 있다.

## 설계 원칙 / 특징

**1. 에이전트 분리 등록.** 기존 스킬은 에이전트 정의를 매 호출마다 프롬프트에 복붙했다. 이 플러그인판은 에이전트를 `agents/`에 정식 등록해, 오케스트레이터가 **이름(`novel-writer:에이전트명`)으로만 호출**한다 → 정의 자동 주입 · 컨텍스트 절약 · 재사용 · 에이전트별 `model`/`tools` 고정.

**2. 판단은 LLM, 조립은 코드.** 검수 반영(어디를 어떻게 고칠지)은 오케스트레이터가 `Edit`으로 처리하지만, 수만 자 합본을 만드는 결정적 조립은 `build_novel.py`가 한다 — LLM 출력 토큰을 쓰지 않는다.

**3. 메인 로깅.** 각 에이전트 호출이 끝날 때마다 오케스트레이터가 결과의 `usage`·`agentId`를 `.agent_log.md`에 한 줄씩 기록한다. 토큰·도구 횟수·소요 시간은 에이전트 자신이 모르므로 메인이 뽑아 적는다.

**4. 진행 배너 + 폴더 안내.** 긴 자동 작업 중 사용자가 현재 위치를 알 수 있게 단계마다 배너를 띄우고, 끝나면 `0_먼저읽으세요.md`로 산출물을 안내한다.

**5. 파일 기반 격리 + 복원형.** 에이전트끼리 직접 통신하지 않고 파일로만 정보를 공유한다(각자 독립 컨텍스트). 산출물이 디스크에 남으므로, 한 워커가 멈춰도 누락된 화만 다시 호출하면 된다.

## 설치

플러그인 마켓플레이스 또는 플러그인 설정에서 이 플러그인을 설치한 뒤, 새 대화에서 `"소설 써줘"`로 트리거한다.

> 플러그인을 새로 설치/갱신하면 보통 **새 대화부터** 반영된다. 진행 중이던 세션엔 이전 버전이 남아 있을 수 있다.

## 제약 / 주의

- **에피소드 수**: 1~5화 권장. 초과 시 컨텍스트 소비가 커진다.
- **에이전트 정의 복붙 금지**: 반드시 등록 이름(`subagent_type`)으로 호출한다(이 플러그인의 설계 목적).
- **`story-bible.md` 필수**: Phase 2 시작 전 반드시 존재해야 한다 — 모든 집필·검수 에이전트가 읽는 유일한 공유 레일.
- **동기 병렬 한계**: 한 워커가 멈추면 전원 완료 대기로 블록될 수 있다. 이미 생성된 파일은 남으니, 중단 후 누락분만 다시 호출한다.
- **무인 자동 아님**: 오케스트레이터가 Phase 사이를 지휘한다(완전 무인 파이프라인이 아니라 단계별 진행).

## 버전 0.4.3 변경점

- 등록 에이전트 **4 → 5** (`character-actor` 추가) + **연기 패스(STEP 2.5)** 도입(recast/핑퐁 2모드).
- `build_novel.py` 조립 파일 감지 픽스 — `ep[0-9][0-9].md`로 한정하고 `*.orig.md`(연기 패스 백업)를 제외.
- 실행 로그를 숨김 파일 `.agent_log.md`로 분리(최종 결과물 옆에 노출 안 됨).
- 단계별 **진행 배너** 규칙 명문화.
- **폴더 안내 `0_먼저읽으세요.md`** 자동 생성(있는 파일만 골라 설명·삭제 가능 표시).
- `references/harness.md` 최신화(character-actor·STEP 2.5 반영).
